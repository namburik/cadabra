paytek: From Blog to Code

A production-grade guide to agentic enterprise systems — every design decision made, every edge case handled, all running together.

May 2026 Reference guide By Kishore Namburi

paytek is a production agentic platform for enterprise merchant onboarding — purpose-built to surface every constraint, trade-off, and failure mode that real systems encounter. This page is the reference: each pattern from this site mapped to the part of paytek that implements it, so the path from concept to working code is always one click away.

The platform

A merchant chats — the AI extracts business details, validates them in real time, and routes through a multi-agent workflow that handles approvals, flags risk, and tracks state across Stripe, PayPal, and Adyen. The architecture makes explicit choices: LangGraph for stateful workflow, MCP for tool grounding, Neo4j for relational context, Postgres and DynamoDB for persistence, a framework-agnostic agent mesh for orchestration, and AI from Google, Ollama, and OpenRouter. Every choice is documented — so when you face the same decision, you know why this one was made.

Architecture

Architecture

The Unified Agentic Stack: A Complete Blueprint for Production AI

paytek is the unified stack — end to end. MCP server for tool grounding, LangGraph for stateful workflow, a multi-framework agent mesh, triple datastore (Postgres + DynamoDB + Neo4j), FastAPI routes, and a Streamlit conversational UI all wired together and running in production on AWS EC2.

Architecture

Beyond GraphRAG 2.0: The Context Infrastructure Mandate

A GraphRAGHandlers class loads the merchant knowledge graph from Neo4j on startup and exposes similarity search as MCP tools. The Neo4j layer uses the FIBO financial ontology as its schema — grounding every agent decision in a structured, queryable knowledge graph, not raw text retrieval.

Architecture

Why SLMs Are the Agent Runtime of Choice

Model selection is config-driven, not hardcoded. OpenRouter integration provides pluggable model backends. The agent bridge imports ChatOllama, making local SLMs a drop-in runtime — no architecture changes needed to swap models at the edge.

Architecture

The Semantic Sidecar Pattern: Decoupled Grounding for AI Agents

The MCP server is the sidecar — a standalone process that provides semantic context and domain grounding over a clean protocol boundary. Agents from any framework connect to it without coupling to the data layer. A semantic context module and core data module separate what agents can know from what agents can do.

Orchestration

Orchestration

Configurable Multi-Agent Workflows: A YAML-Driven Orchestration

Workflow behavior is fully config-driven. Gate check rules, routing conditions, and agent parameters are declared in configuration — not hardcoded in agent logic. Changing a compliance rule or onboarding gate requires no code deployment, only a config update.

Orchestration

Beyond the Bot: The Agentic Playbook

The merchant workflow graph is the playbook in code: conversational intake → structured field extraction with extended thinking → parallel gate checks → LLM risk scoring (0–1 with reasoning) → human-in-the-loop interrupt → persistence. Every step is a named node in the state machine, checkpointed to DynamoDB.

Orchestration

Multi-Agent Systems: The Architecture of Partnership

Four agent frameworks coexist in the same system: LangGraph for stateful process flows, AutoGen for multi-model deliberation, CrewAI for role-based safety gates, and AWS Strands for event-driven tasks. A single bridge module is the only import point — swapping or adding a framework touches one file.

Data & Data Architecture

Data

Agentic AI: Revolutionizing Data Workflows

The entire merchant lifecycle is an agentic data workflow — no manual hand-offs. A LangGraph state machine takes a merchant from natural-language conversation through structured extraction, compliance gate checks, risk scoring, and DB persistence, with full state recovery if any step fails mid-run.

Data Architecture

AI First Series: Building the Data Framework

The data layer is designed for agents to query, not just humans. Domain models backed by FIBO financial ontology, repository and service layers abstracted across Postgres (workflow state), DynamoDB (checkpoints and metadata), and Neo4j (merchant knowledge graph) — each chosen for what it does best.

Data

mytake: How GraphRAG Grounds AI in Knowledge Graph

A sync worker mirrors merchant data from Postgres into Neo4j continuously. A graph service layer exposes traversal and similarity queries. GraphRAG tools surface those queries as MCP tools agents can call — so when the risk scorer evaluates a new merchant, it retrieves the five most structurally similar historical cases as grounded context.

Operations

Operations

From Prototype to Production: Operationalizing AI Agents

The system runs containerised on AWS EC2: Docker + docker-compose, a production startup script with timestamped log rotation, health checks for every service, auto-recovery for stateless services (MCP server restarts safely; Postgres and Neo4j are flagged for human review). A lightweight response evaluator scores agent output quality in real time — not post-hoc.

Governance & Security

Governance

The Negotiator: Why Data Governance Must Stop Saying "No"

Gate checks are configurable rules that approve, escalate, or flag — not hard blocks. The workflow routes around gate failures gracefully, escalating to human review rather than rejecting outright. Governance is a participant in the workflow, not a wall in front of it.

Security

Beyond the Hype: Why AI TRiSM is the Secret to Scaling Enterprise AI

The Guardian agent is the TRiSM layer — a runtime integrity controller that intercepts every write operation before it reaches the database. It validates enterprise identifiers, checks intent against allowed patterns, and blocks mutations that fail the safety check. Enabled by default; bypassing it requires an explicit override, not an omission.

The throughline

None of these patterns exist in isolation. Each was designed to work with the others — and paytek is where they converge into a system you can understand, adapt, and extend. Start with the unified stack overview, then follow any thread that interests you through to the code.
