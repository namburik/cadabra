paytek: Where Agentic AI Meets Enterprise Reality

May 2026 Reference guide By Kishore Namburi

paytek is a agentic platform for enterprise merchant onboarding — built under the constraints, trade-offs, and failure modes that surface in realtime, not in theory. Each capability earned its place independently before the platform took shape. This page maps every pattern from this site to the a working engine that powers it, one click away.

The platform

A merchant chats — an Autonomous agent extracts business details, validates them in real time, and routes through a multi-agent workflow that handles approvals, flags risk, and tracks state across Stripe, PayPal, and Adyen. The architecture makes explicit choices: LangGraph for stateful workflow, MCP for tool grounding, Neo4j for relational context, Postgres and DynamoDB for persistence, a framework-agnostic agent mesh for orchestration, and AI from Google, Ollama, and OpenRouter. Every choice is documented — so when you face the same decision, you know why this one was made.

Architecture

Architecture

Full-stack agentic system

paytek delivers every layer of the unified stack in a single running system — no gaps, no stub implementations. MCP for tool grounding, LangGraph for stateful workflow, a multi-framework agent mesh, triple datastore (Postgres + DynamoDB + Neo4j), FastAPI routes, and a Streamlit conversational UI — all production-deployed on AWS. Can be forked as a foundation.

Architecture

Knowledge-grounded decisions

paytek grounds agent decisions in structured knowledge, not probabilistic text retrieval. The FIBO financial ontology schemas the Neo4j layer, and GraphHandlers exposes it as MCP tools — giving every agent multi-hop reasoning over verified merchant relationships. Decisions backed by facts, not best-guess embeddings.

Architecture

Model-agnostic runtime

paytek makes model selection a config change, not a code change. OpenRouter provides pluggable backends and the ChatOllama bridge makes any local SLM a drop-in runtime — run cost-efficient specialist models at the edge without touching agent logic. Swap, benchmark, or specialise models independently of the rest of the system.

Architecture

Decoupled knowledge layer

paytek's MCP server gives agents a clean protocol boundary to the data layer — any framework connects without coupling to storage. The separation of semantic context from core data means the knowledge layer evolves independently of agent logic. Agents always have a well-defined contract for what they can know and what they can do, with no leakage between the two.

Orchestration

Config-driven workflow logic

paytek separates workflow logic from code entirely — gate rules, routing conditions, and agent parameters live in configuration, not in the agent. Update a compliance rule or gate without touching the codebase. Teams managing policy and teams managing code work independently of each other.

Orchestration

Stateful, recoverable workflow

paytek turns the entire merchant journey into a stateful, recoverable workflow — conversational intake → structured extraction with extended thinking → parallel gate checks → LLM risk scoring (0–1 with reasoning) → human-in-the-loop interrupt → persistence. Every step is named, observable, and checkpointed to DynamoDB. No silent failures, no lost state.

Orchestration

Framework-agnostic agent mesh

paytek runs four agent frameworks in the same system — LangGraph, AutoGen, CrewAI, and AWS Strands — without locking into any one of them. A single bridge module is the only import point, so swapping or extending the agent layer touches one file. The system gets stronger with each framework added, not harder to maintain.

Data & Data Architecture

Data

Zero hand-off merchant lifecycle

paytek eliminates manual hand-offs from the merchant lifecycle entirely. A LangGraph state machine carries every merchant from natural-language conversation through extraction, compliance checks, risk scoring, and persistence — with full state recovery if any step fails mid-run. The workflow either completes or recovers; it never silently drops.

Data Architecture

Agent-first data layer

paytek's data layer is built for agents to query, not just humans. FIBO-backed domain models give agents a shared semantic vocabulary, and the storage layer assigns each concern to the database best suited to it — Postgres for workflow state, DynamoDB for checkpoints, Neo4j for the merchant knowledge graph. No single datastore asked to do everything.

Data

Context-aware risk scoring

paytek gives every risk decision real context, not just the current merchant's data. A continuous sync worker keeps Neo4j current with Postgres, and GraphRAG tools surface traversal and similarity queries as MCP calls — so every risk evaluation is informed by the five most structurally similar historical cases. Decisions improve with every merchant processed.

Operations

Operations

Production-grade deployment

paytek ships production-ready — Docker + docker-compose on AWS, timestamped log rotation, per-service health checks, and auto-recovery built in from the start. Stateless services restart automatically; stateful ones surface for human review rather than failing silently. A lightweight response evaluator scores agent output quality in real time, so issues surface during the run — not in the post-mortem.

Governance & Security

Governance

Governance as workflow participant

paytek makes governance a participant in the workflow, not a gatekeeper in front of it. Configurable rules approve, escalate, or flag — never hard-block. When a check fails, the workflow routes to human review rather than rejecting outright, so compliance teams get oversight without slowing every merchant down.

Read the pattern →
Security

Runtime security enforcement

paytek enforces security at runtime, not at the perimeter. The Guardian agent intercepts every write before it reaches the database — validating enterprise identifiers, checking intent against allowed patterns, and blocking non-compliant mutations on the spot. Enabled by default; a bypass requires an explicit override, never an omission.
